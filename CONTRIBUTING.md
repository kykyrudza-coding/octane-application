## Як повідомити про баг

1. Перевірте [Issues](../../issues) — можливо, баг вже відомий.
2. Відкрийте новий Issue, обравши шаблон **Bug Report**.
3. Вкажіть кроки для відтворення, очікувану і фактичну поведінку, версії PHP та залежностей.

---

## Як запропонувати функцію

1. Відкрийте Issue з шаблоном **Feature Request**.
2. Поясніть проблему, яку вирішує функція, і як ви бачите її реалізацію.
3. Дочекайтеся обговорення перед тим, як писати код.

## Налаштування середовища

```bash
git clone https://github.com/kykyrudza/octane.git
cd octane

composer install
npm install

cp .example.env .env
# налаштуйте .env

php octane serve
```

Для роботи з фронтендом:

```bash
npm run dev
```

## Стиль коду

Проєкт використовує [Laravel Pint](https://github.com/laravel/pint) для форматування PHP-коду.

Перед комітом запустіть:

```bash
./vendor/bin/pint
```

Перевірка без змін:

```bash
./vendor/bin/pint --test
```

Загальні правила:
- PHP 8.2+, строга типізація (`declare(strict_types=1)` там, де це доречно)
- Коментарі — тільки якщо пояснюється **чому**, а не **що**
- Назви класів — `PascalCase`, методів і змінних — `camelCase`
- Нові можливості мови (readonly, enum, match, named arguments) — вітаються

## Процес Pull Request

1. Зробіть форк репозиторію і створіть гілку від `main`:
   ```bash
   git checkout -b feature/my-feature
   # або
   git checkout -b fix/bug-description
   ```

2. Внесіть зміни. Кожен коміт — одна логічна одиниця роботи.

3. Переконайтеся, що код відформатований (`./vendor/bin/pint`).

4. Відкрийте Pull Request на `main`, заповнивши шаблон.

5. Дочекайтеся рев'ю. Може знадобитися внести правки.

**Правила PR:**
- Один PR — одна задача
- Не включайте незв'язані зміни
- Якщо PR вирішує Issue, вкажіть `Closes #N` в описі

## Конвенція комітів

Використовуємо формат [Conventional Commits](https://www.conventionalcommits.org/):

```
<тип>(<область>): <короткий опис>
```

| Тип | Коли використовувати |
|---|---|
| `feat` | Нова функція |
| `fix` | Виправлення бага |
| `refactor` | Рефакторинг без змін поведінки |
| `docs` | Зміни в документації |
| `test` | Додавання або зміна тестів |
| `chore` | Обслуговування (залежності, конфіг тощо) |
| `perf` | Покращення продуктивності |

**Приклади:**

```
feat(router): add PUT, PATCH, DELETE methods
fix(model): use static binding in initialize()
docs(readme): add cache examples
refactor(auth): remove command pattern, inline logic
```
